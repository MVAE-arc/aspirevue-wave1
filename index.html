THE WHY

An Arc'kin Agency is a small, disciplined system.

One operator. A named roster of specialist personas, each with a lane. A vault that holds the memory across sessions. A routing discipline that keeps the right specialist on the right question. That's it. No dashboard. No subscription layer. No product to buy.

The reason it exists: most people using AI are scattered across tools, contexts, and sessions that don't talk to each other. Every prompt starts from zero. Every insight gets retyped. Every decision that compounded on Monday is gone by Friday. You already know this because you've been building the inverse on your own side of this.

Arc'kin is that instinct, turned into a repeatable operating system — with one rule that doesn't bend: the AI does not originate. It does not ideate without instruction, decide without scope, or architect without authorization. Every creative and strategic direction comes from the operator. SOPs, scorecards, and style gates carry that intent into the execution layer so it doesn't get lost in translation. Personas where chat would drift. Memory where sessions would reset. Taste gates where quality would slip. A boss who still decides.

The system is the leverage. The operator stays the operator.

THE ARCHITECTURE

The team.

Ten positions. One boss. Four lanes — Strategy, Build, Ops, Life. Every engagement also gets a dedicated account steward — that's the last row, and it's engagement-specific, not a standing seat.

How it actually moves.

The boss talks to ORIENT. ORIENT routes. Most work runs as a swarm: a question about positioning pulls STRATEGOS and RECON together; a landing page pulls CANVAS, SIGNAL, and FORGE in parallel; a voice note gets caught by CIRCUIT and handed to whoever owns the subject. Agents collaborate directly across lanes when the work calls for it.

ANCHOR sits alongside ORIENT, not inside the work stack. Its job is to keep the life side visible while the work side compounds, so the operator doesn't accidentally trade one for the other.

ATLAS joins on a per-engagement basis. For AspireVue, ATLAS keeps the promise ledger, the cadence, the running memory of every exchange. It's a quiet function. You notice it when something you mentioned four weeks ago shows up in a draft without you having to remind anyone.

THE WORKFLOW

How the work actually moves.

Three input channels.

Local

Typed prompts at the terminal. Voice-to-text via native dictation. Files dragged directly into a session as the assignment — no explanation needed, the drop is the ask.

Remote

Claude remote-control for taking over a live session from anywhere. Claude Dispatch for quick retrievals that don't need a conversation. Plaud transcripts routing through automation into agent pickup, so a voice memo in the car becomes a primed prompt by the time the laptop opens.

Exterior

Google Antigravity summoning and triggering sessions externally. Automations that fire scheduled work. Remote triggers from calendar, inbox, or vault events.

Three doors, same vault.

Claude Code (via Antigravity)

The execution surface. Full Claude Max capacity. Reads and writes files directly. Spawns specialist agents in parallel, which is how the swarming pattern actually runs. Where things happen.

Claude Cowork

Project-scoped chat that mirrors the Code environment. Same vault, same agents, different register. Quieter. Useful when a session wants room to think but still needs to write home.

Claude chat (desktop/app)

The familiar interface. Best for ideation, early shaping, refining a brief before it enters the working system. When a spec is ready, it moves to Obsidian as a note, and the next Code session opens on that note.

THE MEMORY LAYER

Obsidian holds everything.

Every agent, every SOP, every decision, every piece of engagement memory lives in the vault as plain markdown. Folders for active projects. Folders for personas. A MOC for the memory index. Date-stamped notes for sessions and handoffs. Nothing proprietary to a cloud app. Nothing locked behind a subscription that could deprecate under you.

The reason this matters is practical. Continuity across sessions: when the next Code session opens, the relevant files are already the context, not an upload ritual. Token economy: you pay for signal, not for reintroducing yourself to the system. Organizational sanity: if a decision was authorized, it's in a file with a date; if it wasn't, you can see that too.

The vault looks spare on purpose. No dashboards, no color themes, no cover images. It is a working surface. The aesthetic is legibility under load, not demo polish. When you're in it at five in the morning, you want it to get out of your way.

APPENDIX · THE HARDWARE

What the system runs on.

The rig isn't the point. But for a reader who thinks about infrastructure at the same resolution as the software, the hardware is the receipt. Six pieces, each with a job and a reason.

The capture device — Plaud
A small voice recorder that clips to a lapel or sits on the desk. Transcribes with Apple-native share-out, which is the friction difference between a voice note that becomes an agent input and one that stays on a phone. Voice is the highest-bandwidth input humans produce. The Plaud turns that bandwidth into operable text without a typing layer in between.
The compute pair — Mac Mini M1 (primary) + MacBook Pro 2017 (mobile mirror)
A fixed desktop is the primary. It holds the vault, runs the sessions, never moves. The laptop is a mirror — same vault via iCloud, same sessions via remote control, used when the work travels. Splitting primary from mobile is deliberate: a single laptop forced to be both burns out, disorients, and loses discipline. Two machines, one of them stationary, preserves the separation between deep-work environment and travel surface.
The display rig — Samsung CHG90 49" ultrawide + Asus ZenScreen Touch + iPad Pro (Sidecar)
Parallel agent sessions want parallel windows. The CHG90 is effectively two 1920-wide monitors in one surface, which is the minimum real estate for running four or more windows without cycling focus. The ZenScreen adds touch/pen input for design work. iPad Pro via Sidecar adds a fifth window when the session demands it. Window count carries the swarming pattern physically — the desk is the diagram.
The notes layer — reMarkable (V1)
An e-ink tablet for handwritten thinking. Offline by design. Used when a problem wants longhand pacing before it's ready for a session. Pairs with Plaud for thought-tracking during reviews and document reads. The reason it stays in the stack is that some problems get smaller when typed and some get sharper when drawn.
The storage philosophy — G-Tech G-RAID (enterprise RAID) + SanDisk 4TB SSD + 1TB SSD
SSD only. No spinning disks anywhere in the pipeline. Enterprise-grade RAID for business-critical backup. The RAID could house the full vault independent of the M1 if needed. The rule is simple: reliability first, cost second. Losing a client vault to a failed drive is a category of mistake the system refuses to make possible.
The mobile — iPhone 16 Pro Max
Plaud companion. Claude remote-control surface. Calendar, inbox, the rest. The phone is the thinnest part of the stack because the heavy work should not happen on it. When a prompt is firing from the phone, the goal is fewest keystrokes possible — voice-first, file-drop second, text as a fallback.
